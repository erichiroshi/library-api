# PostgreSQL + Flyway + Schema per Service — Library API

## O que este documento cobre

Como o banco de dados foi estruturado no projeto: versionamento de schema com Flyway, isolamento de domínios com Schema per Service, o decrement atômico para controle de cópias, auditoria automática, e como essa estrutura facilita a transição para microservices.

---

## PostgreSQL no Projeto

O PostgreSQL é o banco principal do projeto. Todas as entidades dos três bounded contexts (auth, catalog, lending) vivem nele — separadas por schemas, não por bancos distintos.

A escolha do PostgreSQL sobre alternativas como MySQL tem razões práticas:

- Suporte nativo a schemas (namespaces dentro do mesmo banco)
- `GENERATED BY DEFAULT AS IDENTITY` para PKs (padrão SQL, mais portável que `SERIAL`)
- Suporte robusto a tipos, índices parciais e `UPDATE ... WHERE` atômico
- Integração madura com Spring Data JPA e Testcontainers

---

## Flyway: Versionamento de Schema

Flyway é uma ferramenta de migração de banco de dados. A ideia central é simples: cada mudança no schema é um arquivo SQL numerado e versionado. O Flyway controla quais migrações já foram aplicadas e executa apenas as novas.

### Por que isso importa em produção

Sem Flyway (ou similar), o fluxo comum é:
- Desenvolvedor altera uma entidade JPA
- `ddl-auto: update` tenta sincronizar o banco automaticamente
- Em produção, isso é perigoso — o Hibernate pode dropar colunas, recriar tabelas, perder dados

Com Flyway:
- `ddl-auto: validate` — o Hibernate apenas verifica se o schema bate com as entidades. Se não bater, a aplicação não sobe.
- Cada mudança é explícita, rastreada, versionada e reversível (com scripts de rollback)
- O histórico do banco acompanha o histórico do código no Git

### Estrutura das migrações

```
src/main/resources/db/migration/
├── common/          ← aplicadas em todos os profiles
│   ├── V001__create_tables_category_author_book_book_author.sql
│   ├── V002__create_tables_user_loan_loan_item.sql
│   ├── V003__alter_table_user.sql
│   ├── V004__create_refresh_token_and_adjust_columns.sql
│   ├── V005__add_audit_and_version_columns.sql
│   ├── V006__add_cover_image_url_tb_book.sql
│   ├── V007__create_shedlock_table.sql
│   ├── V008__create_schemas.sql
│   └── V009__migrate_tables_to_schemas.sql
└── dev/
    └── V1000__dev_seed_data.sql   ← dados de teste, só em dev
```

O prefixo numérico garante a ordem de execução. O Flyway aplica sempre em sequência crescente e nunca re-executa uma migração já aplicada (rastreia via tabela `flyway_schema_history`).

### Convenção de nomenclatura

```
V{versão}__{descrição}.sql

V001__create_tables_...    ← versão 1, underscores duplos obrigatórios
V1000__dev_seed_data       ← número alto garante que dev seed roda por último
```

### Separação common/dev

Migrações em `common/` rodam em todos os ambientes. O seed de dados (V1000) fica em `dev/` e só é carregado quando o profile `dev` está ativo:

```yaml
# application-dev.yml
flyway:
  locations:
    - classpath:db/migration/common
    - classpath:db/migration/dev   # ← só em dev

# application-prod.yml
flyway:
  locations:
    - classpath:db/migration/common  # ← sem dados de teste em prod
  clean-disabled: true               # ← proíbe flyway:clean em prod
```

`clean-disabled: true` é uma proteção importante — o comando `flyway:clean` dropa todas as tabelas. Em produção, isso seria catastrófico.

## A Evolução do Schema: 9 Migrações

As migrações contam a história do projeto:

**V001 e V002** — estrutura inicial
Tabelas principais criadas: `tb_category`, `tb_author`, `tb_book`, `tb_book_author`, `tb_user`, `tb_loan`, `tb_loan_item`. Neste ponto tudo estava no schema `public`, sem separação de domínios.

**V003** — senha e roles
Adicionada coluna `password` ao `tb_user` e criada `tb_user_roles`. Mostra como Flyway lida com evolução incremental — não é necessário recriar a tabela, apenas alterar.

**V004** — refresh tokens
Criação de `tb_refresh_tokens` com índices para performance: `idx_refresh_token_user` (otimiza `findByUser`) e `idx_refresh_token_expiry` (otimiza a query de limpeza de tokens expirados).

**V005** — auditoria
Adicionadas colunas de auditoria em todas as tabelas: `created_at`, `updated_at`, `created_by`, `last_modified_by`, `version`. Feito via `ALTER TABLE` — sem perda de dados existentes.

**V006** — URL de capa
Coluna `cover_image_url` adicionada ao `tb_book` para suportar upload via S3.

**V007** — ShedLock
Tabela `shedlock` para lock distribuído dos scheduled jobs. Fica no schema `public` intencionalmente — é infraestrutura, não domínio de negócio.

**V008 e V009** — Schema per Service
As migrações mais importantes do projeto. V008 cria os schemas, V009 move as tabelas:

```sql
-- V008
CREATE SCHEMA IF NOT EXISTS auth;
CREATE SCHEMA IF NOT EXISTS catalog;
CREATE SCHEMA IF NOT EXISTS lending;

-- V009
ALTER TABLE tb_user           SET SCHEMA auth;
ALTER TABLE tb_user_roles     SET SCHEMA auth;
ALTER TABLE tb_refresh_tokens SET SCHEMA auth;

ALTER TABLE tb_book        SET SCHEMA catalog;
ALTER TABLE tb_author      SET SCHEMA catalog;
ALTER TABLE tb_category    SET SCHEMA catalog;
ALTER TABLE tb_book_author SET SCHEMA catalog;

ALTER TABLE tb_loan      SET SCHEMA lending;
ALTER TABLE tb_loan_item SET SCHEMA lending;
```

`ALTER TABLE ... SET SCHEMA` move a tabela sem recriar — dados preservados, FKs mantidas, sem downtime.

## Schema per Service

### O problema que resolve

Em microservices, a regra é clara: cada serviço é dono exclusivo dos seus dados. Nenhum outro serviço acessa diretamente o banco de outro.

Se tudo estivesse no schema `public`, misturado, a extração futura exigiria:
- Identificar quais tabelas pertencem a qual serviço
- Mover dados entre bancos
- Atualizar FKs e referências
- Coordenar downtime

Com schemas separados, a fronteira já existe. A migração futura é só uma questão de apontar cada serviço para seu próprio PostgreSQL.

### Como funciona hoje

O HikariCP é configurado com `search_path` para resolver tabelas automaticamente:

```yaml
datasource:
  hikari:
    connection-init-sql: SET search_path TO auth,catalog,lending,public
```

Isso significa que uma query como `SELECT * FROM tb_user` é resolvida para `auth.tb_user` automaticamente. As entidades anotam o schema explicitamente como reforço:

```java
@Table(name = "tb_user",   schema = "auth")
@Table(name = "tb_book",   schema = "catalog")
@Table(name = "tb_loan",   schema = "lending")
```

Para tabelas de junção e coleções:

```java
@JoinTable(
    name = "tb_book_author",
    schema = "catalog",           // ← schema explícito
    joinColumns = @JoinColumn(name = "book_id"),
    inverseJoinColumns = @JoinColumn(name = "author_id")
)

@CollectionTable(
    name = "tb_user_roles",
    schema = "auth",              // ← schema explícito
    joinColumns = @JoinColumn(name = "user_id")
)
```

### A regra de ouro: sem FK entre schemas

Não existe nenhuma FK cruzando schemas no projeto. Isso é intencional e fundamental.

Por exemplo: `tb_loan` (lending) referencia `tb_user` (auth) e `tb_book` (catalog). Mas essa referência é mantida via código, não via constraint de banco:

- `LoanService` busca o usuário pelo email via `UserLookupService`
- `LoanService` busca o livro pelo id via `BookLookupService`
- O banco não tem FK de `lending.tb_loan` para `auth.tb_user`

Isso pode parecer uma fragilidade — e tecnicamente é, pois o banco não garante integridade referencial entre domínios. Mas é a mesma situação que existirá em microservices, onde cada serviço tem seu próprio banco e integridade referencial entre serviços é responsabilidade da aplicação.

## Decrement Atômico de Cópias

Esta é uma das decisões mais interessantes do projeto — resolver um problema de concorrência sem locks explícitos.

### O problema

Dois usuários tentam emprestar o último exemplar de um livro ao mesmo tempo:

```
Usuário A: lê availableCopies = 1 → vai decrementar
Usuário B: lê availableCopies = 1 → vai decrementar
Usuário A: UPDATE SET availableCopies = 0 ✓
Usuário B: UPDATE SET availableCopies = 0 ✓  ← race condition!
```

Ambos conseguem emprestar, mas só havia 1 cópia. O resultado: `availableCopies = -1`.

### A solução

```java
@Modifying(clearAutomatically = true, flushAutomatically = true)
@Query("""
    UPDATE Book b
    SET b.availableCopies = b.availableCopies - 1
    WHERE b.id = :id
    AND b.availableCopies > 0
""")
int decrementCopies(@Param("id") Long id);
```

O banco executa o UPDATE atomicamente. A condição `AND b.availableCopies > 0` garante que o UPDATE só ocorre se ainda há cópias. O retorno é o número de linhas afetadas:

- `1` → decrement realizado com sucesso
- `0` → não havia cópias disponíveis

```java
int updated = bookAvailabilityPort.decrementCopies(bookId);
if (updated == 0) {
    throw new BookNotAvailableException(bookId, book.getTitle());
}
```

Com dois usuários concorrentes:
- O banco processa os dois UPDATEs em sequência (serialização interna)
- O primeiro recebe `rowsUpdated = 1`
- O segundo recebe `rowsUpdated = 0` (pois `availableCopies` já é 0)
- O segundo lança `BookNotAvailableException`

### Por que clearAutomatically e flushAutomatically?

```java
@Modifying(clearAutomatically = true, flushAutomatically = true)
```

**`flushAutomatically = true`**: antes de executar o UPDATE, garante que operações pendentes (dirty checking) são enviadas ao banco. Evita que um insert ou update anterior ainda em memória interfira com a ordem de execução.

**`clearAutomatically = true`**: após o UPDATE, limpa o cache de primeiro nível do JPA (EntityManager). Sem isso, uma chamada subsequente a `findById()` na mesma transação retornaria o valor antigo em memória, ignorando o UPDATE recém-executado no banco.

## Auditoria Automática com Spring Data JPA

Toda entidade herda de `BaseEntity`:

```java
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @CreatedDate
    private LocalDateTime createdAt;

    @LastModifiedDate
    private LocalDateTime updatedAt;

    @CreatedBy
    private String createdBy;

    @LastModifiedBy
    private String lastModifiedBy;

    @Version
    private Long version;
}
```

O Spring Data preenche automaticamente:
- `createdAt` / `updatedAt` — timestamps gerenciados pelo framework
- `createdBy` / `lastModifiedBy` — preenchidos pelo `AuditorAware`
- `version` — controle de concorrência otimista

### AuditorAware: quem está operando

```java
@Bean
AuditorAware<String> auditorProvider() {
    return () -> Optional.ofNullable(
        SecurityContextHolder.getContext().getAuthentication()
    ).map(Authentication::getName);
}
```

O `createdBy` e `lastModifiedBy` são preenchidos com o email do usuário autenticado no momento da operação — lido diretamente do `SecurityContext`, sem query adicional.

### @Version: Optimistic Locking

O campo `version` implementa controle de concorrência otimista. Quando dois processos tentam atualizar a mesma entidade simultaneamente:

```
Processo A: lê versão 1 → modifica → tenta salvar com versão 1
Processo B: lê versão 1 → modifica → tenta salvar com versão 1
Processo A: salva → banco atualiza versão para 2 ✓
Processo B: tenta salvar versão 1 → banco tem versão 2 → OptimisticLockException ✗
```

O banco rejeita o segundo UPDATE porque a versão não bate. Isso é mais eficiente que locks pessimistas para operações de baixa contenção.

## Índices Estratégicos

Os índices criados nas migrações têm justificativas específicas:

```sql
-- Otimiza findByUser() no RefreshTokenRepository
CREATE INDEX idx_refresh_token_user ON tb_refresh_tokens(user_id);

-- Otimiza DELETE FROM tb_refresh_tokens WHERE expiry_date < NOW()
-- Query executada todo dia pelo RefreshTokenCleanupJob
CREATE INDEX idx_refresh_token_expiry ON tb_refresh_tokens(expiry_date);

-- Garante unicidade do token
CREATE UNIQUE INDEX idx_refresh_token_unique ON tb_refresh_tokens(token);

-- Otimiza queries de busca por URL de capa
CREATE INDEX idx_book_cover_image_url ON tb_book(cover_image_url);
```

### Trade-off de índices

Índices aceleram leituras mas têm custo em escritas — cada INSERT ou UPDATE precisa atualizar os índices relevantes. A estratégia é criar índices apenas onde há queries frequentes ou críticas, não em todas as colunas.

## Configuração JPA/Hibernate

```yaml
jpa:
  open-in-view: false          # desabilita anti-pattern de sessão aberta na view
  hibernate:
    ddl-auto: validate         # só valida — nunca altera o schema
  properties:
    hibernate:
      '[default_schema]': public
      '[jdbc.batch_size]': 20  # batch inserts/updates
      '[order_inserts]': true  # agrupa inserts por tabela para maximizar batching
      '[order_updates]': true  # agrupa updates por tabela
```

**`open-in-view: false`**: por padrão, o Spring Boot mantém a sessão JPA aberta durante toda a renderização da resposta HTTP, permitindo lazy loading fora da transação. Isso é um anti-pattern — esconde N+1 queries e cria acoplamento entre camadas. Com `false`, qualquer acesso lazy fora da transação lança exceção imediatamente, forçando o carregamento explícito dentro da transação.

**`batch_size: 20`**: ao invés de 20 INSERTs individuais, o Hibernate agrupa em um batch e envia ao banco de uma vez. Reduz drasticamente o número de round-trips em operações em massa.

## Fase 3: O Banco na Transição para Microservices

### O que muda

Hoje: um PostgreSQL, três schemas, uma connection pool.

Futuro: três PostgreSQL independentes, um por serviço.

```
HOJE:
┌─────────────────────────────────┐
│ PostgreSQL                      │
│   schema: auth                  │
│   schema: catalog               │
│   schema: lending               │
└─────────────────────────────────┘

FUTURO:
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ PostgreSQL   │  │ PostgreSQL   │  │ PostgreSQL   │
│ (auth)       │  │ (catalog)    │  │ (lending)    │
└──────────────┘  └──────────────┘  └──────────────┘
```

### Como a migração acontece na prática

1. Criar três instâncias PostgreSQL (Docker, RDS, etc.)
2. Para cada serviço, copiar as tabelas do schema correspondente para o novo banco
3. Atualizar as configurações de datasource de cada microservice
4. Remover o `search_path` multi-schema — cada serviço conhece apenas seu banco

As Flyway migrations de cada serviço começam do zero — apenas as tabelas do domínio correspondente. O histórico do monolito fica no monolito.

### O problema das transações distribuídas

No monolito, criar um empréstimo e decrementar as cópias do livro acontecem na **mesma transação ACID**:

```
@Transactional
public LoanResponseDTO create(LoanCreateDTO dto) {
    decrementCopies(bookId);   // catalog schema
    save(loan);                // lending schema
    // se qualquer coisa falhar aqui, tudo reverte
}
```

Com bancos separados, não existe transação distribuída nativa. Se o `catalog-service` decrementar as cópias e o `loan-service` falhar ao salvar o empréstimo, os dados ficam inconsistentes.

**Solução: Saga Pattern**

A Saga decompõe a transação em etapas com compensações:

```
1. loan-service: cria empréstimo com status PENDING
2. loan-service: publica evento "decrementar cópia do livro X"
3. catalog-service: tenta decrementar
   ├── sucesso: publica "cópia decrementada"
   │   → loan-service: atualiza status para CONFIRMED
   └── falha: publica "decrement falhou"
       → loan-service: cancela empréstimo (compensação)
```

Isso é significativamente mais complexo que a transação ACID atual. É um dos principais trade-offs da transição para microservices.

### O que o projeto já tem que ajuda

- **Schema per Service**: fronteiras de dados já explícitas, sem FK entre domínios
- **Sem FK entre schemas**: sem constraint no banco que impeça a separação
- **LookupServices**: a camada de abstração já existe, só muda a implementação
- **Spring Events com primitivos**: estrutura dos eventos é compatível com Kafka/RabbitMQ

## Resumo das Decisões de Banco de Dados

| Decisão | Alternativa | Por que esta |
|---|---|---|
| Flyway com `ddl-auto: validate` | `ddl-auto: update` | Controle explícito, seguro em produção, histórico rastreável |
| Schema per Service no mesmo banco | Banco separado por domínio desde o início | Fronteiras sem overhead operacional prematuro |
| Sem FK entre schemas | FK cruzando schemas | Força isolamento real, compatível com separação futura |
| Decrement atômico via UPDATE condicional | SELECT + UPDATE separados ou lock explícito | Evita race condition sem overhead de lock pessimista |
| `open-in-view: false` | Padrão Spring Boot (true) | Evita N+1 oculto, força carregamento explícito na transação |
| `@Version` optimistic locking | Lock pessimista | Mais eficiente para baixa contenção |
| Índice em `expiry_date` do refresh token | Sem índice | Query de cleanup diário sem full table scan |
| Seed separado em `dev/` | Seed em `common/` | Dados de teste nunca chegam em produção |
